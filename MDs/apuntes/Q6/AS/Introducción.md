A arquitectura do software é a **descomposición** do software nun conxunto de **compoñentes/módulos** e **iteracción** entre eles. A definición da arquitectura dun software consiste en realizar a **selección** deses elementos que o integran e decidir a súa **organización**. A arquitectura do software debe permitir satisfacer os **requisitos non funcionais** do software, e sirva de base para o seu posterior deseño e implementación.

- O deseño software
	- Deriva dos **requisitos funcionais**
	- Organizámolo usando **patróns de deseño**
	- Pode despregarse en diferentes arquitecturas
- A arquitectura do software
	- Deriva dos **requisitos non funcionais**
	- Organizámolo usando **estilos** de arquitectura.

Definir a arquitectura é tomar as **decisións de máis impacto** que dan forma ao software, onde o impacto se mide en **custe de cambio**.
- Estructura, tecnoloxías, requisitors non funcionais.

Os **estilos de arquitectura** determinan o uso de certos elementos e o seu xeito de organizarse.
- Serven como mecanismo de comunicación entre desenvolvedores, ao igual que os patróns de deseño software.

A arquitectura do software represéntase habitualmente de xeito informal e heteroxéneo. Existe un estándar, o IEEE 1471, pouco adotado. Calquera representación debe:
- Indicar que é cada compoñente e o papel que xoga.
- Mostrar as interaccións entre compoñentes e o seu fin.
- Manter coherencia entre compoñentes semellantes.
- Manter coherencia entre interaccións equivalentes.

**Modelo C4**: a arquitectura do software represéntase mediante unha sucesión de catro vistas.
- **Contexto**, que mostra a relación do sistema con outros
- **Contedor**, que mostra a distribución física dos compoñentes.
- **Compoñente**, que mostra as relacións entre compoñentes.
- **Código** (*Diagramas de compoñentes UML*).
